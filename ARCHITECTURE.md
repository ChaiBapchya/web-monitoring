## System Architecture

This section briefly describes components of the system.

<a id="orgae876cd"></a>

### Page Collection

We will be using the PageFreezer harvesting software until at least 2020, thanks to their generous donation. PageFreezer crawls government webspace frequently (~once/week) and records any changes it sees. This takes place via several instances of PageFreezer, installed on EDGI&rsquo;s cloud computing resources (generously donated by Google). We don&rsquo;t yet know the mechanics of storage, but differences between page versions are available via an API call, and for now, at least, we have access to full pages as well. **While we rely on PageFreezer, this is proprietary software, and we will not be modifying it in any way. The crawling & harvesting are, therefore, out of scope for our project**

Interfaces:

-   READS from `Storage` while crawling, to check for differences (via the `diffing service`?)
-   WRITES to `Storage` when new versions are found

<a id="org5cbd32e"></a>

### Storage

HTML and other resources are stored in Google cloud storage, along with an accompanying database that tracks changes. Database schemas are not currently known.

Interfaces:

-   Communicates with `Page Collection` as above
-   Provides responses to `Diffing Service` queries about individual pages (see below)
-   Provides lists of pages and versions to `Researcher UI` (see below)

<a id="orgba1a9c9"></a>

### Diffing Service

This service accepts queries about individual pages (and perhaps large sets of pages) and returns diffs in JSON format. We can use PageFreezer&rsquo;s existing service or create a replacement, depending on how closely our needs are matched by PageFreezer&rsquo;s. [We already have a candidate](https://github.com/edgi-govdata-archiving/differ) (developed to overcome some shortcomings of PageFreezer&rsquo;s *external* diff service), but we should probably evaluate the more robust internal diff UI with which we will soon be provided.

Interfaces:

-   Queries `Storage` DB (via SQL?)
-   Provides JSON object to `UI`

<a id="org0ee60b9"></a>

### Filtration Service

This service uses a set of heuristics to filter changes out of the result set presented in the UI. The heuristics will include generic and domain-specific rules. Eventually we would like to replace the ad-hoc rules in this service with machine-learning algorithms.

Interfaces:

-   Either queries Storage for new pages or receives notification from Page Collection  when new pages are added to Storage.
-   Assigns a priortiy or other labels that can be used to filter important/unimportant changes and, ideally, prioritize them by estimated importance.

<a id="org3b5f7b7"></a>

### User Interface

This web-based interface presents end users (analysts) with views of page collections (e.g., all pages from a given domain that pass the `filtration Service` tests, above) and individual pages. Analysts will be able to tag particular changes as belonging to a variety of categories, to be used later in EDGI&rsquo;s weekly reports. We&rsquo;ll place a high premium on ease of use, and speed of movement through changed pages; we&rsquo;ll probably want to be at least somewhat clever in our use of prefetching, etc.

Interfaces:

-   Requests individual page diffs from `diff service`
-   Requests full lists of changed pages from `Storage`
-   Writes results of analysts&rsquo; inspection to `monitoring-database`

<a id="org1c8db9d"></a>

### Monitoring Database

Storage of analysts&rsquo; classification of page diffs. Question: is this the same as `Storage`? What are the advantages & disadvantages of keeping them separate or unified?

Interfaces:

-   Communicates with `User Interface`, above
-   Can be queried through some other mehcanism to produce reports?

## Interfaces

Interface contract template:

| Characteristic | Description                                 |
|----------------|---------------------------------------------|
| *Frequency*    | Indicate approximately how often this interface accessed, e.g., once a week, 500 times a day, etc. |
| *Timing*       | List and specific time requirements |
| *Sizing*       | Indicate expected transfer rates, e.g., must be able to load X records per minute <br /> Indicate number of records per transfer <br /> Indicate estimated size of record (e.g. 100 bytes, 10K, …) |
| *Format*       | Describe in detail the format of the transfer, e.g., XML DTDs, file record layouts, etc. The information in this section will vary depending on the interface specifics |

| *Other*        | Indicate any other "good things to know."  |

### `Diffing Service`  &harr; `UserInterface`

| Characteristic | Description                                 |
|----------------|---------------------------------------------|
| *Frequency*    | ~100-500x per user session (lasting ~1-2 hours) |
| *Timing*       | Help with this? |
| *Sizing*       | Record size is ~100-300kb. # of records/request TBD -- currently single records, but batching may be more efficient |
| *Format*       | HTML get query UI &rarr; DS (see below);  JSON back (see [note](#notes) for an example) |
| *Security*     | Help with this? |
| *Other*        |  |

Accepts a query with the following parameters:

| Parameter               | Description                        |
|-------------------------|------------------------------------|
| `source` (optional)     | Default: `url`. <br>~url~=url1 and url2 must be URL of the target document. <br>~text~=url1 and url2 contains HTML text document itself. |
| `url1`                  | The source URL or HTML                                                                                     |
| `url2`                  | The target URL or HTML                                                                                     |
| `diffmode`              | (optional)	Default: `0`. <br>~0~=No pre-processing, <br>~1~=extra white spaces removed, <br>~2~=[\s]\* are removed,<br>~3~=HTML tags are removed for full-text comparison |
| `html`  (optional)      | Default: `1`. <br> `2` = HTML with HEAD, <br> `1` =HTML without HEAD, <br> `0` = False (no HTML output).   |
| `snippet`  (optional) | Default: `200` (characters). It will generate snippets of changes.                                           |

Returns a JSON Object. Full spec is not known but an example is given in [Notes](#notes), below.
